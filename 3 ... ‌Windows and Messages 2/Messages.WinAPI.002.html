Common Windows Messages: Your App's Communication Manual

Message Meaning / When It's Sent (The Vibe) What You Usually Do (The Move)
WM_CREATE This message pops up right when your window is getting ready to exist, just after it's been created. Think of it as the "setup" notification. This is your go-to spot to set up all the initial stuff for your window: load any pictures or sounds, kick off timers, or get your app's starting game pieces ready.
WM_DESTROY Your window is officially signing off! This message hits when the window is being totally removed, like when a user clicks the 'X' or your code explicitly destroys it. When you get this, it's the signal to gracefully shut down. Most importantly, call PostQuitMessage(0) here to tell your main message loop that it's time to pack up and exit the app.
WM_PAINT Windows is giving you a heads-up: "Hey, part of your window needs a fresh coat of paint!" This happens when a window becomes visible, is uncovered, or when you explicitly ask it to redraw. Inside this message, you usually do your drawing. You'll kick things off with BeginPaint, use the provided drawing context (HDC) to paint your stuff, and then wrap it up with EndPaint. No drawing outside of this!
WM_QUIT This is a special internal "game over" message. Your main message loop (GetMessage) uses this to know when to finally stop running and let your application close. You actually **don't** handle this one directly in your WndProc function. Instead, GetMessage just returns 0 when it sees this, which is the signal for your program's main loop to exit. It's the loop's secret handshake to say "we're done."
WM_CLOSE Someone just clicked the 'X' button on your window, or hit Alt+F4! This is their polite way of saying, "Can I close this window, please?" You can catch this message if you want to ask the user, "Are you sure you want to quit?" before letting the window close. If they say no, you can stop it; otherwise, let DefWindowProc handle it.
WM_COMMAND This is like a notification from your app's controls or menus. It fires when someone clicks a button, picks an item from a menu, or interacts with other UI elements. Check wParam (specifically its lower 16 bits) to figure out *which* specific menu item or control sent this message. That's how you know what action to take!
WM_KEYDOWN A key on the keyboard just got pressed down. This message fires when any key is physically pushed. Use wParam to see *which* key was pressed. It'll be a virtual key code (like VK_ESCAPE for the Escape key or VK_LEFT for the left arrow). Great for game controls or shortcuts!
WM_CHAR This message shows up **after** WM_KEYDOWN, but only if you've called TranslateMessage in your message loop. It means a *character* has been typed. If you're building a text input box or need to capture actual letters, numbers, or symbols (like 'a', 'Z', '7', '!'), this is the message you'll use. wParam will hold the ASCII or Unicode value of the character.
WM_KEYUP The opposite of WM_KEYDOWN: a key on the keyboard was just released. You might not always need this, but it's handy for things like detecting when a 'shift' key is released, or when you want to stop an action that started on key down.
WM_LBUTTONDOWN Boom! The left mouse button was clicked while the mouse pointer was inside your window's client area (the part you can draw on). This is your cue for single clicks, starting a drag-and-drop operation, or initiating any action tied to a left mouse press.
WM_LBUTTONUP The left mouse button, which was pressed, has now been released. This usually marks the end of a click action or the completion of a drag-and-drop. You often pair this with WM_LBUTTONDOWN.
WM_MOUSEMOVE The mouse pointer just moved a tiny bit while it was over your window's client area. Get ready for a lot of these if the user is fidgety! Great for tracking the cursor's position, creating hover effects (like highlighting a button when the mouse is over it), or drawing lines when dragging.
WM_SIZE Your window just got a makeover â€” someone resized it! It fires whenever the window's dimensions change. This is where you'd adjust the layout of your elements inside the window. If you have buttons, text boxes, or images, this is where you'd make sure they reposition or resize correctly to fit the new window size.
WM_TIMER Tick-tock! This message is sent to your window after a specific amount of time has passed, if you've set up a timer using SetTimer. Perfect for anything that needs to happen regularly, like animations, checking for updates, or polling for certain conditions.
WM_SETCURSOR The mouse pointer is hovering over your window, and Windows is asking, "What kind of cursor should I show here?" Use this to customize the mouse cursor when it's over different parts of your window â€“ maybe a hand over a button, or an I-beam over text.
WM_HOTKEY You pressed a specific keyboard shortcut that you've registered system-wide (meaning it works even when your app isn't the active window). This is how you handle global shortcuts in your app, like 'Ctrl+Alt+S' to save from anywhere, or 'Win+Z' to open your app.

ðŸ’¡ The Golden Rule: Don't Leave Windows Hanging!

If your WndProc function doesn't specifically handle a message (because you only care about certain ones), it's super important to let Windows take over. Think of it like this: if you, the chef, don't know how to make a dish, you pass it to the head chef.

Always return this if you don't handle a message:

return DefWindowProc(hwnd, message, wParam, lParam);

This way, Windows handles all the default behavior for you, like closing windows, moving them around, minimizing, maximizing, and all that good stuff without you having to code it!
