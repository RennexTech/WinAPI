Windows Messages & Message Loop – The Core Story

Concept What it Does (The Vibe)
Message Think of it like a text message from Windows or your user saying, "Hey, something just happened!" Could be a key press, a mouse click, or Windows asking your app to redraw itself.
Message Queue This is like your app's inbox. When Windows has a bunch of messages for your app, it lines them up here, waiting for your app to come check them out.
MSG structure This is the actual message package. It's got all the deets: which window it's for (hwnd), what kind of message it is (message), and extra info (wParam, lParam) that changes based on the message. Like a delivery label with specifics.
GetMessage(&msg,...) This function is like your app constantly checking its inbox for new messages. It grabs the next one in line. If it sees a "Time to peace out!" message (WM_QUIT), it signals that it's done.
TranslateMessage(&msg) Ever type a key and want to make sure it's treated as a character (like 'A' instead of just "key down")? This function is the translator. It takes a raw key press (WM_KEYDOWN) and, if needed, generates a character message (WM_CHAR). Super useful for text input!
DispatchMessage(&msg) After getting and maybe translating a message, this is like the app's internal postman. It takes the message and delivers it straight to the specific window's "brain" (WndProc) that needs to handle it.
WndProc(...) This is your app's main decision-maker, your custom message router. When a message lands here, this is where you write the code to decide what your window should actually *do* with that message. Like a central command center.
WM_PAINT This message is Windows tapping your app on the shoulder, saying, "Hey, your window needs a fresh coat of paint!" It means a part of your window is now visible or needs to be redrawn.
WM_DESTROY This is the "It's time to close up shop" message for a window. It's sent when a window is about to be completely removed from the screen. Usually, right after this, you tell the whole app to shut down with PostQuitMessage.
PostQuitMessage(0) This is like hitting the "all-clear" button for your app. It sends a special WM_QUIT message to the message queue, which is the signal for the main message loop to finally stop running.
WM_QUIT The ultimate "I'm out!" message. When GetMessage() sees this one in the queue, it knows the party's over and returns 0, making your app's main loop exit gracefully.
wParam / lParam These are like extra notes attached to the message. The meaning of what's inside them totally depends on what kind of message it is. Could be anything from a key code to mouse coordinates.
POINT pt Tucked inside the MSG structure, this little gem tells you exactly where the mouse was chilling on the screen when that specific message was sent. Super useful for mouse interactions!
Loop Lifetime This message-checking process keeps going, non-stop, until that special WM_QUIT message shows up in the queue, signaling GetMessage() to finally return 0 and break the cycle.
TranslateMessage needed? Only if you're building something where users will actually type stuff, like a text box or an input field, and you need to process actual characters, not just raw key presses. Otherwise, you can skip it.
Multiple windows? No sweat! Every single message comes with a return address – the HWND (Handle to Window) – which tells your app exactly *which* window the message is meant for. So, messages always go to the right place.
